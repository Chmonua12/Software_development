## Выводы

### Пропускная способность

На маленьких сообщениях (128B-1KB) RabbitMQ показывает более высокую пропускную способность (414-444 msg/s против 331-382 msg/s у Redis). На больших сообщениях (10-100KB) Redis неожиданно оказывается быстрее (425-429 msg/s против 337-350 msg/s у RabbitMQ). Это может быть связано с особенностями реализации асинхронного consumer в тесте.

### Влияние размера сообщения

При увеличении размера сообщения с 128B до 100KB пропускная способность падает у обоих брокеров:
- RabbitMQ: с ~430 до ~210 msg/s (падение на 51%)
- Redis: с ~430 до ~240 msg/s (падение на 44%)

Задержки P95 растут:
- RabbitMQ: с 3.2 до 7.0 мс (+119%)
- Redis: с 2.3 до 7.0 мс (+204%)

Redis показывает более низкие задержки на маленьких сообщениях, но при росте размера задержки увеличиваются быстрее.

### Точка деградации single instance

При текущей конфигурации теста (реальная скорость producer ~350-440 msg/s) ни один из брокеров не достиг точки деградации:
- Потери сообщений отсутствуют (lost=0)
- Задержки остаются низкими (1-7 мс)
- Очередь не накапливается (backlog=0)

Для достижения точки деградации необходимо увеличить реальную скорость отправки до 5000-10000 msg/s, что требует оптимизации producer или использования многопоточности.

### Какой брокер лучше

**Для сценариев с низкой нагрузкой (до 500 msg/s):**
- Оба брокера работают отлично, потерь нет
- Redis даёт чуть меньшие задержки
- RabbitMQ проще в настройке для очередей

**Для высоконагруженных сценариев:**
- По литературным данным и предыдущим тестам, RabbitMQ лучше справляется с высокими нагрузками (10000+ msg/s)
- Redis без Streams теряет сообщения при переполнении очереди

**Рекомендация:** Для production-сценариев с гарантированной доставкой выбирайте RabbitMQ. Для простых кэш-сценариев с низкой нагрузкой подойдёт Redis.
