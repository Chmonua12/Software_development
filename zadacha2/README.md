# Сравнение RabbitMQ и Redis в роли брокеров сообщений (Практическая работа №2)

## Архитектура стенда

Стенд включает три логические части:

| Компонент | Функция |
|-----------|---------|
| **producer** | Формирует сообщения нужного размера и отправляет в брокер с заданной интенсивностью |
| **broker** | Поочерёдно тестируются RabbitMQ (классическая очередь с подтверждением) и Redis (режим очереди через списки) |
| **consumer** | Принимает сообщения, подтверждает обработку и собирает метрики |


## Матрица экспериментов

Параметры прогонов (в точности по заданию):

| Параметр | Значения |
|----------|----------|
| Размер сообщения | 128 B, 1 KB (1024 B), 10 KB (10240 B), 100 KB (102400 B) |
| Интенсивность потока | 1 000 msg/s, 5 000 msg/s, 10 000 msg/s |
| Длительность одного прогона | 10 секунд |

**Итого:** 2 брокера × 4 размера × 3 интенсивности = **24 тестовых прогона**.

## Итоговые выводы

### Пропускная способность

- **RabbitMQ** показал значительно более высокую пропускную способность. На сообщениях 128–1024 байта он обрабатывает ~875–945 msg/s, тогда как Redis — только ~145–350 msg/s.
- С ростом размера сообщений разрыв увеличивается:
  - При 10 KB: RabbitMQ ~663–711 msg/s, Redis ~90–106 msg/s
  - При 100 KB: RabbitMQ ~407–432 msg/s, Redis ~56–99 msg/s

### Влияние размера сообщения

- У обоих брокеров увеличение payload снижает throughput, но **Redis деградирует сильнее** по скорости обработки.
- Задержки растут с увеличением размера у обоих. На 100 KB p95 достигает десятков секунд у обоих брокеров.

### Точка деградации single instance

Признаки деградации: рост backlog и резкое увеличение p95 задержек.

- **RabbitMQ:** заметный backlog начинает появляться при размере 10 KB (backlog 17–40), p95 задержка растёт до ~14–15 секунд.
- **Redis:** признаки деградации видны уже на маленьких сообщениях при высоком rate — количество принятых сообщений значительно ниже отправленных. На больших payload скорость обработки падает до десятков msg/s.

### Выбор инструмента для нагрузочного тестирования

В работе использован **собственный Python-runner** по следующим причинам:

- Обеспечивает **строго одинаковый формат сообщений** для RabbitMQ и Redis
- Позволяет вычислять **end-to-end задержку** по временной метке `sent_ts_ns`
- Даёт полный контроль над метриками и подтверждением обработки
