# Сравнение RabbitMQ и Redis в роли брокеров сообщений (Практическая работа №2)

## Архитектура стенда

Стенд включает три логические части:

| Компонент | Функция |
|-----------|---------|
| **producer** | Формирует сообщения нужного размера и отправляет в брокер с заданной интенсивностью |
| **broker** | Поочерёдно тестируются RabbitMQ (классическая очередь с подтверждением) и Redis (режим очереди через списки) |
| **consumer** | Принимает сообщения, подтверждает обработку и собирает метрики |


## Матрица экспериментов

Параметры прогонов (в точности по заданию):

| Параметр | Значения |
|----------|----------|
| Размер сообщения | 128 B, 1 KB (1024 B), 10 KB (10240 B), 100 KB (102400 B) |
| Интенсивность потока | 1 000 msg/s, 5 000 msg/s, 10 000 msg/s |
| Длительность одного прогона | 10 секунд |

**Итого:** 2 брокера × 4 размера × 3 интенсивности = **24 тестовых прогона**.

## Итоговые выводы

### Пропускная способность

- **RabbitMQ** показал значительно более высокую пропускную способность. На сообщениях 128–1024 байта он обрабатывает ~875–945 msg/s, тогда как Redis — только ~145–350 msg/s.
- С ростом размера сообщений разрыв увеличивается:
  - При 10 KB: RabbitMQ ~663–711 msg/s, Redis ~90–106 msg/s
  - При 100 KB: RabbitMQ ~407–432 msg/s, Redis ~56–99 msg/s

### Влияние размера сообщения

- У обоих брокеров увеличение payload снижает throughput, но **Redis деградирует сильнее** по скорости обработки.
- Задержки растут с увеличением размера у обоих. На 100 KB p95 достигает десятков секунд у обоих брокеров.

### Точка деградации single instance

Признаки деградации: рост backlog и резкое увеличение p95 задержек.

- **RabbitMQ:** заметный backlog начинает появляться при размере 10 KB (backlog 17–40), p95 задержка растёт до ~14–15 секунд.
- **Redis:** признаки деградации видны уже на маленьких сообщениях при высоком rate — количество принятых сообщений значительно ниже отправленных. На больших payload скорость обработки падает до десятков msg/s.

### Выбор инструмента для нагрузочного тестирования

В работе использован **собственный Python-runner** по следующим причинам:

- Обеспечивает **строго одинаковый формат сообщений** для RabbitMQ и Redis
- Позволяет вычислять **end-to-end задержку** по временной метке `sent_ts_ns`
- Даёт полный контроль над метриками и подтверждением обработки


## Ответы на вопросы 

### 1. Какой брокер показал большую пропускную способность?

**RabbitMQ.** Он справляется с нагрузкой гораздо лучше. Например, на сообщениях 1 КБ при интенсивности 1000 msg/сек RabbitMQ обработал около 9700 сообщений из 10000, а Redis только около 2700. Разница примерно в 3-4 раза в пользу RabbitMQ. На высоких нагрузках разрыв становится ещё больше.

### 2. Какой брокер лучше работает с маленькими сообщениями?

**RabbitMQ** тоже лучше. На сообщениях 128 байт он показывает лучшую скорость обработки и меньшие задержки. Но важно другое: Redis на маленьких сообщениях теряет очень много данных — около 70-80% сообщений просто не доходят до потребителя, потому что очередь переполняется. RabbitMQ теряет не больше 2-3%.

### 3. Какой брокер быстрее деградирует на больших сообщениях?

**Redis деградирует намного быстрее.** Когда размер сообщения достигает 100 КБ, Redis обрабатывает только 16-22 сообщения в секунду, а RabbitMQ — 400-450. У Redis пропускная способность падает в 10-20 раз при увеличении размера с 1 КБ до 100 КБ. У RabbitMQ падение примерно в 2-3 раза.

### 4. При какой нагрузке начал деградировать RabbitMQ?

Заметная деградация RabbitMQ началась при размере сообщения 10 КБ и интенсивности 5000-10000 msg/сек. В этот момент:
- Задержка p95 выросла до 14-15 секунд (вместо 0.2-0.4 секунды на маленьких сообщениях)
- Появились первые потери сообщений
- Пропускная способность упала примерно на 30-40%

При 100 КБ деградация стала критической — задержки достигают 30 секунд.

### 5. При какой нагрузке начал деградировать Redis?

Redis начал деградировать **сразу на первой же нагрузке**. Уже при 1000 msg/сек и размере 128 байт он теряет около 70% сообщений. При увеличении интенсивности до 5000-10000 msg/сек ситуация становится ещё хуже — обрабатывается только 2-3% сообщений, остальные теряются. Задержки при этом достигают 20 секунд.

Фактически Redis в такой схеме использования (LPUSH/BRPOP) не подходит для сколько-нибудь серьёзной нагрузки.

### 6. Какой брокер в итоге лучше для реального использования?

**RabbitMQ.** Он:
- Обрабатывает в 3-10 раз больше сообщений
- Теряет гораздо меньше данных
- Предсказуемо работает на разных размерах сообщений
- Нормально держит нагрузку до 5000-10000 msg/сек

Redis без дополнительных настроек (как Redis Streams) для очередей не подходит — слишком много потерь даже на маленькой нагрузке.

### 7. Какой инструмент выбрали для тестирования и почему?

Использовали свой скрипт на Python. Почему:
- Можно сделать одинаковый формат сообщений для обоих брокеров
- Легко считать точную задержку (отправка → получение)
- Просто добавить новые размеры и интенсивности
- Не нужно настраивать сложные инструменты типа JMeter

Минус — сложно выдерживать точную интенсивность (RPS может плавать), но для сравнения двух брокеров в одинаковых условиях это не критично.
