## Описание проекта

**Цель**: разработать backend-ориентированное приложение для арт-комьюнити с многоуровневой системой рейтинга анкет и современной, масштабируемой архитектурой.

Приложение должно:
- **Формировать рейтинг анкет** на основе анкетных данных и поведения пользователей.
- **Обрабатывать события взаимодействия** (лайки, переходы по ссылкам, регистрации) в потоковом режиме.
- **Обеспечивать высокую производительность** за счёт кэширования топа, отложенных пересчётов и очередей сообщений.
- **Интегрироваться с внешними соцсетями** (Telegram, Instagram, VK, Behance) через кнопки-ссылки.

Проект предполагается как набор микросервисов (или модульных bounded context'ов) с чёткими зонами ответственности.

---

## Сервисы и их ответственность

- **Auth / Users Service**
  - Регистрация и аутентификация пользователей.
  - Хранение основных данных пользователя (Telegram ID, базовые настройки).
  - Управление сессиями/токенами доступа.
  - Реферальная система (генерация ссылок, учёт приглашённых).

- **Profile Service (Анкеты)**
  - Хранение расширенной информации анкеты:
    - имя, возраст, город, геолокация;
    - описание творчества, интересы (теги/категории);
    - предпочтения (кого ищет, возрастной диапазон, город/радиус).
  - Управление фотографиями анкет (метаданные + ссылки на S3/Minio).
  - Оценка **полноты заполнения профиля** и количества фотографий (вклад в первичный рейтинг).
  - Управление социальными сетями:
    - хранение ссылок (Telegram, Instagram, VK, Behance, другая);
    - валидация формата ссылок;
    - приоритет Telegram как основной соцсети.

- **Rating Service (Рейтинги)**
  - Реализация **трёхуровневой модели рейтинга**:
    - Уровень 1: первичный рейтинг (качество анкеты).
    - Уровень 2: поведенческий рейтинг (динамика взаимодействий).
    - Уровень 3: комбинированный рейтинг (интеграция + рефералы).
  - Хранение агрегированных рейтингов в отдельной таблице/схеме БД.
  - Регулярные пересчёты рейтингов по расписанию через **Celery** (полный пересчёт раз в час).
  - Инкрементальное обновление поведенческого рейтинга при лайке/переходе.
  - Интеграция с очередями сообщений (Kafka) для обработки событий взаимодействия.

- **Interaction Service (Взаимодействия / Лайки / Переходы)**
  - Обработка действий:
    - лайк анкеты;
    - пропуск анкеты;
    - переход по ссылке на соцсеть.
  - Генерация событий в MQ (`profile_liked`, `profile_skipped`, `social_link_clicked`).
  - Ведение статистики лайков/скипов/переходов для расчёта поведенческого рейтинга.
  - Логирование переходов по ссылкам для аналитики.

- **Feed / Recommendation Service (Лента анкет)**
  - Формирование списка анкет для показа пользователю:
    - с учётом фильтров (возраст, город/радиус);
    - с учётом комбинированного рейтинга.
  - Интеграция с **Redis**:
    - кэширование топ-10 художников (обновление каждые 5 минут);
    - предварительное формирование отсортированных списков анкет;
    - при старте сессии:
      - первая анкета рассчитывается «вживую»;
      - параллельно в Redis подгружается пачка следующих N анкет;
      - по мере «выработки» пачки — фоновая догрузка следующей.

- **Media Service (Хранение изображений)**
  - Загрузка и управление изображениями анкет (аватарка + до 5 работ).
  - Интеграция с **S3-совместимым хранилищем (Minio)**:
    - генерация presigned URL для загрузки/просмотра;
    - хранение только ссылок и метаданных в основной БД.

- **Notification Service (Уведомления)**
  - Отправка уведомлений пользователям:
    - когда их лайкнули;
    - когда попали в топ;
    - о новых рефералах.
  - Интеграция с Telegram Bot API для отправки сообщений.

- **Metrics / Logging / Monitoring**
  - Сбор технических и бизнес-метрик:
    - количество лайков/скипов/переходов;
    - время ответа ключевых эндпоинтов;
    - количество активных пользователей;
    - статистика по пересчётам рейтингов.
  - Централизованное логирование запросов и фоновых задач (Loki).
  - Метрики через Prometheus.

## Архитектура и взаимодействие компонентов

### Общий обзор

- **Клиенты** (Telegram) обращаются к **Bot Service** (aiogram).
- Bot Service взаимодействует с **API Gateway** (FastAPI).
- Backend разбит на доменные сервисы (описаны выше).
- Для обмена событиями между сервисами используется **Kafka**.
- Асинхронные и периодические задачи выполняются через **Celery** (worker'ы).
- **Redis** используется как быстрый кэш для:
  - топ-10 художников;
  - предварительно отсортированных списков анкет;
  - сессий/токенов (опционально).
- **S3/Minio** для файлов (изображения анкет).
- Основная **реляционная БД** (PostgreSQL) для транзакционных данных.

### Поток взаимодействия (упрощённо)

1. **Регистрация/авторизация**:
   - Пользователь запускает бота (`/start`).
   - Bot Service создаёт/находит пользователя в `Auth Service` по Telegram ID.
   - Создаётся базовая запись профиля в `Profile Service`.

2. **Заполнение анкеты**:
   - Пользователь заполняет анкету, загружает фото (через `Media Service` → S3/Minio).
   - Пользователь добавляет соцсети (Telegram, Instagram, VK, Behance) — бот валидирует ссылки.
   - `Profile Service` сохраняет данные и рассчитывает степень заполненности профиля.
   - `Rating Service` формирует **первичный рейтинг (Уровень 1)** на основе:
     - возраста, города, описания творчества, интересов;
     - полноты анкеты и количества фото;
     - бонуса за Telegram как основную соцсеть.

3. **Формирование ленты анкет**:
   - Пользователь открывает ленту художников.
   - `Feed Service` запрашивает текущий кэш в **Redis**:
     - если кэш пуст или устарел, запрашивает `Rating Service`/`Profile Service` для генерации свежего списка.
     - первая анкета вычисляется онлайн, параллельно в Redis подгружается пачка следующих N анкет.

4. **Лайки/пропуски/переходы**:
   - Пользователь лайкает или пропускает анкету через `Interaction Service`.
   - При нажатии на кнопку соцсети — переход по ссылке, событие логируется.
   - `Interaction Service`:
     - записывает событие в БД;
     - публикует событие в Kafka (`profile_liked`, `profile_skipped`, `social_link_clicked`).

5. **Обновление поведенческого рейтинга (Уровень 2)**:
   - `Rating Service` подписан на события Kafka.
   - На основе:
     - количества лайков и их отношения к просмотрам;
     - количества переходов по ссылкам на соцсети;
     - временных паттернов активности;
   - рассчитывается **поведенческий рейтинг**.
   - Инкрементальное обновление — при каждом лайке/переходе.
   - Тяжёлые перерасчёты — через Celery.

6. **Комбинированный рейтинг (Уровень 3)**:
   - В отдельной задаче (через Celery по расписанию, раз в час) `Rating Service`:
     - интегрирует **первичный** и **поведенческий** рейтинги по весовой модели (40% + 60%);
     - учитывает **реферальную активность** (бонусы за приглашённых друзей).
   - Результат записывается в таблицу рейтингов.
   - Кэш Redis обновляется (топ — каждые 5 минут, лента — по событию/TTL).

7. **Топ художников**:
   - `Feed Service` запрашивает топ-10 из Redis (обновляется каждые 5 минут).
   - Команда `/top` возвращает список с кнопками соцсетей.

## Схема данных (черновой вариант)

Ниже — логическая модель основных сущностей (без конкретного синтаксиса SQL).

- **Пользователь и профиль**
  - `users`
    - `id`
    - `telegram_id` (unique)
    - `created_at`
    - `status` (active, banned, deleted)
  - `profiles`
    - `id`
    - `user_id` (FK → `users.id`)
    - `display_name`
    - `age`
    - `city`
    - `location_lat`, `location_lon` (опционально)
    - `bio` (описание творчества)
    - `profile_completeness_score` (0–1 или 0–100)
    - `photos_count`
    - `created_at`, `updated_at`
  - `profile_interests`
    - `id`
    - `profile_id` (FK)
    - `interest_tag` (строка или FK в `interests`)
  - `profile_preferences`
    - `id`
    - `profile_id` (FK)
    - `age_min`, `age_max`
    - `city` или `radius_km`

- **Социальные сети**
  - `profile_social_links`
    - `id`
    - `profile_id` (FK)
    - `platform` (telegram, instagram, vk, behance, other)
    - `url`
    - `is_primary` (флаг основной соцсети)
    - `created_at`

- **Фото**
  - `profile_photos`
    - `id`
    - `profile_id` (FK)
    - `storage_key` (путь/ключ в S3/Minio)
    - `url` (опционально, presigned)
    - `is_avatar` (флаг аватарки)
    - `order_index`
    - `created_at`

- **Взаимодействия**
  - `profile_likes`
    - `id`
    - `from_profile_id`
    - `to_profile_id`
    - `created_at`
  - `link_clicks`
    - `id`
    - `profile_id` (чью ссылку открыли)
    - `viewer_profile_id` (кто открыл)
    - `platform` (какую соцсеть)
    - `created_at`

- **Рейтинги**
  - `profile_ratings`
    - `id`
    - `profile_id` (FK, unique)
    - `primary_rating` (Уровень 1)
    - `behavior_rating` (Уровень 2)
    - `combined_rating` (Уровень 3)
    - `likes_count`
    - `views_count`
    - `link_clicks_count`
    - `referral_score` (баллы за приглашённых друзей)
    - `last_recalculated_at`

- **Рефералы**
  - `referrals`
    - `id`
    - `inviter_profile_id`
    - `invited_user_id`
    - `status` (registered, active, bonus_applied)
    - `created_at`

- **Технические таблицы**
  - `rating_recalc_jobs` — история задач пересчёта рейтингов.
  - `events_log` — лог бизнес-событий.
  - `top_cache` — кэш топ-10 (опционально, если не только Redis).

## Рейтинг: уровни и алгоритмы (концептуально)

- **Уровень 1 — Первичный рейтинг**
  - Основывается на:
    - наличие аватарки;
    - количество фото работ (до 5);
    - полнота описания творчества;
    - указание возраста и города;
    - наличие интересов (тегов);
    - бонус за Telegram как основную соцсеть.
  - Может быть реализован как взвешенная сумма нормализованных факторов.

- **Уровень 2 — Поведенческий рейтинг**
  - Динамически корректируется на основе поведения других пользователей:
    - общее количество лайков анкеты;
    - соотношение лайков к просмотрам;
    - количество переходов по ссылкам на соцсети;
    - временные параметры (в какое время суток анкету чаще лайкают).
  - Все события поступают через Kafka, обрабатываются worker'ами Celery.
  - Инкрементальное обновление — при каждом лайке/переходе.

- **Уровень 3 — Комбинированный рейтинг**
  - Интеграция первичного и поведенческого рейтингов **по весовой модели**:
    - `combined_rating = 0.4 * primary_rating + 0.6 * behavior_rating + referral_score`
  - Учитывает **реферальную программу** (приглашение друзей).
  - Пересчёт по расписанию (раз в час) через Celery.

## Использование Celery, Redis, Kafka и S3 (Minio)

- **Celery**
  - Периодические задачи:
    - массовый пересчёт поведенческого и комбинированного рейтингов (раз в час);
    - обновление топ-10 (каждые 5 минут);
    - очистка устаревших данных в Redis;
    - агрегация статистики.
  - Фоновые задачи:
    - пересчёт рейтинга конкретной анкеты после всплеска активности;
    - обработка тяжёлых запросов.

- **Redis**
  - Кэш топ-10 художников (обновление каждые 5 минут).
  - Кэш предранжированных списков анкет по пользователям/сегментам.
  - Пример:
    - при входе пользователя формируется/обновляется список анкет;
    - первая анкета — онлайн расчёт, параллельно в Redis подгружается пачка из N следующих;
    - при достижении конца пачки — асинхронно формируется следующая.

- **Apache Kafka**
  - Потоковая обработка событий:
    - `profile_liked`, `profile_skipped`, `social_link_clicked`, `referral_registered`.
  - Декуплинг: сервисы лайков, рейтингов и аналитики работают независимо, подписываясь на нужные топики.

- **S3 / Minio**
  - Хранение фотографий:
    - загрузка через presigned URL;
    - проверка/валидация файлов (тип, размер);
    - возможность лёгкой замены Minio на облачное S3.
